Claude Code plugin · fsautocomplete · MIT

You ask in plain language. Claude asks the compiler.

Claude Code ships an LSP tool, but it does not speak F#. That tool is generic — a socket with no server behind it until a plugin registers one for your file extension. Nothing listens on .fs out of the box. There is no official F# plugin. This is that plugin.

You

Ask a question: where is parseWire used? No commands, no coordinates.

Claude

Finds the symbol's position, queries fsautocomplete, and acts on the answer.

Install it and nothing changes that you type. That is the point, and it is why the plugin is easy to overlook: F# code intelligence starts working, and Claude reaches for it on its own. What arrives is goToDefinition, hover, findReferences, documentSymbol, workspaceSymbol, goToImplementation and call hierarchy, on .fs, .fsi and .fsx.

⚠ Ionide in your editor does not carry over

The most common reason people assume they do not need this: F# already works in their editor. It does — for the editor. Claude Code keeps its own LSP client and its own server registry, and a VS Code extension registers with VS Code. The two do not even share a server process: Ionide bundles its own fsautocomplete.dll; this plugin runs the one on your PATH. Here is this plugin disabled, with Ionide 7.31.1 running and healthy on four live FSAC instances, asked for hover on a .fs file:

plugin disabled, Ionide running

/// what this is

Three files, one job each.

Register the server, teach Claude to use it, and speak up when the environment cannot run it.

.lsp.json

Twenty-one lines registering fsautocomplete for .fs, .fsi and .fsx. One warm server per session, answers immediate. You never invoke it.

the skill

Teaches Claude which operation answers which question, how to refactor with no rename available, and why a grep count is not a reference count.

check_fsharp_lsp.py

Runs at session start, silent unless something is wrong. It exists for one failure that reports nothing at all — see below.

/// the loop

One rename, from question to proof.

There is no rename operation. Claude Code's LSP tool is entirely reads — nine operations, not one of them a write. That sounds like a gap, and in F# it very nearly is not, because the compiler closes it at the end.

  1. Every place it is used

    findReferences at the declaration. The answer is the compiler's, so it crosses project boundaries — a symbol in the main project shows its uses in the test project too — and it excludes the comments, strings and longer identifiers a text search would have swept in.

    findReferences — Domain/Status.fs:86:9
    Found 3 references across 3 files:
    Domain/Status.fs
    Line 86:9
    Web/Handlers.fs
    Line 138:38
    Tests/Tests.fs
    Line 277:43

    Exact to the character. Column 38 of Handlers.fs line 138 is where parseWire starts, not where the line does.

  2. Edit each site

    The positions were the hard part of the job and you already have them. This step is mechanical, and it is the one a real textDocument/rename would have done atomically — the only thing actually lost by not having one.

  3. Ask again, and check the old name is gone

    Same position, same question. A symbol that still exists always returns at least its own declaration, so one reference means the declaration and nothing else — which is also how you spot dead code.

  4. Let the compiler have the last word

    This is the step that makes the missing rename tolerable. A site you missed is FS0039 — the value or constructor is not defined — a hard error, with no configuration required to make it one. It cannot pass quietly.

    the last word
    $ dotnet build --no-incremental
    Handlers.fs(138,38): error FS0039: The value or constructor 'parseWire' is not defined.

    --no-incremental is not decoration. A repeat build re-reports nothing, so warnings you have not fixed can look as though they went away. An error also suppresses the warnings in every file compiled after it, since F# stops there.

Manual renaming is far safer in F# than in a dynamically typed language, and this is why: the failure is loud, immediate, and points at the line.

/// install

Two steps to a working setup.

  1. Install the language server

    fsautocomplete is a .NET global tool. Everything depends on it.

    confirm it runs
    $ fsautocomplete --version
    0.83.0+96fabed8e9181b74e19e211717626c204c32b2c2

    Verify rather than trust: the check that matters is whether the binary runs, not whether the install reported success.

  2. Install the plugin

    Then restart. Disabling a plugin drops its server immediately, but enabling one does not bring it back until the session restarts — a difference worth knowing before you conclude the install failed.

⚠ The failure almost everyone hits

~/.dotnet/tools must be on the PATH of the process that launches Claude Code. A login shell that has it is not enough if your editor or launcher does not inherit that PATH — which is why step 1 verifies the binary rather than trusting the install.

/// troubleshooting

Three failures. Only one of them is silent.

Read the message before reaching for anything. Two of the three say what is wrong, and running a health check on them wastes a minute.

what you might see
No LSP server available for file type: .fs
→ no server registered. The plugin is not installed, or not enabled.
Couldn't find <file> in LoadedProjects
→ a server IS running. That file's project is outside the directory
Claude Code was launched in. Open a session there.
(a hang, or silence)
→ registered, but the binary will not start. Almost always PATH.
This is the one the health check exists for.

The third reports nothing because the server dies before the LSP handshake completes, so the tool waits rather than failing. The plugin runs check_fsharp_lsp.py at every session start to catch it, and it prints nothing unless something is wrong. Run by hand it says what it found — this is a working install, with a project directory passed as the optional argument:

a healthy install
$ python3 tools/check_fsharp_lsp.py ~/src/orderboard
ok python 3.10
ok fsautocomplete 0.83.0 (/home/you/.dotnet/tools/fsautocomplete)
ok dotnet sdk 10.0.110
ok 2 restored project(s) under /home/you/src/orderboard

And against a broken one — the binary is installed here, it is simply not on PATH:

a broken install
$ python3 tools/check_fsharp_lsp.py
ok python 3.10
ok dotnet sdk 10.0.110
FAIL fsautocomplete is not on PATH. Install it with 'dotnet tool install -g fsautocomplete', then make sure the global tools directory (~/.dotnet/tools on Linux and macOS) is on the PATH of the process that launches Claude Code. A login shell rc file is not always enough, and that gap is the most common cause of this.
$ echo $?
2

The binary is installed in that run — it is simply not on PATH, and the check refuses to be reassured by finding it elsewhere. Claude Code launches the server as a bare command, so a copy the shell cannot resolve is a copy the server cannot start. It reports every check rather than stopping at the first failure, and it executes fsautocomplete rather than testing for a file, because a binary that exists but dies on startup produces the same silent hang. Pass a project directory as an argument and it also checks that the directory really holds a restored .fsproj.
