Claude Code plugin · fsautocomplete · MIT

You ask in plain language. Claude asks the compiler.

F# code intelligence for Claude Code, so that “find every use of this before I rename it” gets answered by fsautocomplete instead of by a text search.

You

Ask a question: where is double used? No commands, no coordinates.

Claude

Finds the symbol's position, runs the query, reads the output, and acts on it.

A line and column argument is Claude's problem, not yours — and no editor is involved. The tool starts its own fsautocomplete process and takes the project directory as an argument, so nothing needs to be open in VS Code, Rider or anywhere else. Everything below was produced in a bare shell.

“Use when you need a compiler-accurate answer about F# code — every reference to a symbol, the diagnostics for a file, or the symbols a file defines — and especially when the file belongs to a project outside the directory Claude Code was launched in.”

— skills/fsharp-lsp-queries/SKILL.md, the instruction Claude reads

The plugin also installs a session-start hook that runs doctor on its own, so a broken environment announces itself before it can waste your time. Nobody types that one either.

/// behind the scenes

One rename, from question to proof.

This is the whole loop Claude runs for a request like “rename double to twice” — and it is also exactly what you would type to do it yourself. Every command and every line of output below was run against a real F# project; nothing is illustrative.

Setup: a copy of this repository's test project, with fsharp_lsp.py copied in beside it, which is how the CLI is meant to be used. Two files matter — Library.fs defines double, and Consumer.fs calls it.

  1. Is the environment actually usable?

    Nothing else can work if the language server cannot start, and its failure is silent — the built-in tool simply hangs. So the loop opens by executing fsautocomplete rather than trusting that it is installed.

    doctor
    $ python3 fsharp_lsp.py doctor SampleProject
    ok python 3.10
    ok fsautocomplete 0.83.0 (PATH: /home/diego/.dotnet/tools/fsautocomplete)
    ok dotnet sdk 10.0.110
    ok 1 restored project(s) under SampleProject

    Four checks, all green, including that the project directory really holds a restored .fsproj. Any of them failing here would otherwise surface later as a hang with no message.

  2. Where does the symbol live?

    To ask about a symbol you have to point at it, and to point at it you need its line. symbols lists everything a file defines, with the line each one is on — no editor, no scrolling, no guessing.

    symbols
    $ python3 fsharp_lsp.py symbols SampleProject SampleProject/Library.fs
    3 top-level symbol(s):
    SampleProject.Library (line 1)
    answer (line 4)
    double (line 7)

    double is on line 7. The column is where the name itself starts — for a top-level binding that is column 5, immediately after let . That gives the 7 5 the next step needs.

  3. Every place it is used

    Now the real question. The answer comes from the compiler's own view of the program, so it knows the difference between the symbol and a word that merely looks like it.

    references
    $ python3 fsharp_lsp.py references SampleProject SampleProject/Library.fs 7 5
    WITH config: 3 reference(s) in 2 file(s)
    Consumer.fs: 2
    5:26
    5:34
    Library.fs: 1
    7:5

    Three positions: the declaration itself at Library.fs 7:5 — a rename has to touch that too — plus two calls in Consumer.fs, on the same line, at columns 26 and 34. That line reads let quadruple (x: int) = double (double x). A search-and-replace counts that line once; the compiler counts it twice.

  4. Make the edits

    This step is not the tool's job. fsharp_lsp.py is read-only by design — it hands over the positions and the edits stay yours, or Claude's. Two details matter: apply them right to left within a line, so replacing the first call does not shift the column of the second, and verify each position holds the text you expected before touching it.

    after the edits
    $ grep -n twice SampleProject/Library.fs SampleProject/Consumer.fs
    SampleProject/Library.fs:6:/// Referenced twice from Consumer, in one expression.
    SampleProject/Library.fs:7:let twice (x: int) = x * 2
    SampleProject/Consumer.fs:5:let quadruple (x: int) = twice (twice x)
    SampleProject/Consumer.fs:7:let twiceTheAnswer = answer * 2

    Both calls on line 5 were renamed. And notice grep making the case for the whole exercise: two of these four hits are noise — a doc comment that happens to contain the word “twice”, and an unrelated binding called twiceTheAnswer. Neither is the symbol.

  5. Prove nothing broke

    A rename that misses one call site still parses; it fails at the type checker. So the loop closes by asking the compiler directly, which is something the built-in LSP tool cannot do at all.

    diagnostics
    $ python3 fsharp_lsp.py diagnostics SampleProject SampleProject/Consumer.fs
    No diagnostics.

    Clean. Had a call site been missed, this would name it as The value or constructor 'double' is not defined, with its line.

  6. Confirm the loop closed

    Ask the same question again, at the same position, and check that the old name is gone from the project entirely.

    references, again
    $ python3 fsharp_lsp.py references SampleProject SampleProject/Library.fs 7 5
    WITH config: 3 reference(s) in 2 file(s)
    Consumer.fs: 2
    5:26
    5:33
    Library.fs: 1
    7:5
    $ grep -rn double SampleProject --include=*.fs ; echo $?
    1

    Still three references — and the second one moved from column 34 to 33, because twice is one character shorter than double. These are live positions computed from the current source, not remembered from the previous run. grep exits 1: the old name appears nowhere, not even in a comment.

The same loop finds dead code. Run step 3 against quadruple and it returns 1 reference in 1 file, at the position you pointed to — the declaration and nothing else, which means nothing calls it.

/// what this is

A plugin for the workspace. A CLI for everything outside it.

The LSP plugin

Wires fsautocomplete into Claude Code's built-in LSP tool, so navigation works on .fs, .fsi and .fsx — one warm server, fastest for casual work.

The query CLI

Drives its own FSAC process to answer what the built-in tool cannot: any project outside the current workspace, and compiler diagnostics.

  • references
  • diagnostics
  • symbols
  • doctor

“Ask it about a file outside that root and it answers Couldn't find <file> in LoadedProjects, with no way to point it elsewhere. This tool takes the workspace root as an argument, so it can answer questions about any project on disk.”

/// install

Three steps to a working setup.

  1. Install the language server

    fsautocomplete is a .NET global tool. Everything else depends on it.

    confirm it runs
    $ fsautocomplete --version
    0.83.0+96fabed8e9181b74e19e211717626c204c32b2c2

    If that prints nothing, stop here and go to troubleshooting — every later step depends on this one, and its failure is silent.

  2. Install the plugin

    Run both lines inside Claude Code. Click either to copy it.

    This is all you need for navigation inside the project Claude Code was launched in. Step 3 is for the projects it cannot reach.

  3. Copy the CLI into the repository you want to query

    tools/fsharp_lsp.py is a single file with no dependencies beyond Python 3.9+ — standard library only, nothing to pip install. That is the setup used throughout the walkthrough above.

    the four commands
    fsharp_lsp.py doctor [PROJECT] [--hook]
    fsharp_lsp.py references PROJECT FILE LINE COL [--no-config]
    fsharp_lsp.py diagnostics PROJECT FILE [--verbose]
    fsharp_lsp.py symbols PROJECT FILE

/// driving it yourself

The two arguments people get wrong.

PROJECT — the directory FSAC loads

The one holding the .fsproj, which in most solutions is not the repository root. Get this wrong and the server loads nothing, then answers confidently about a workspace it never opened.

FILE — a path from where you are standing

Not relative to PROJECT. This is the mistake everyone makes first, including while this page was being written:

FILE resolved against the wrong root
$ python3 tools/fsharp_lsp.py references tests/fixtures/SampleProject Library.fs 7 5
ERROR: [Errno 2] No such file or directory: 'Library.fs'
$ echo $?
2

Pass the file the way you would to cat, and the two arguments stop looking redundant: PROJECT tells the server what to load, FILE tells it what you are pointing at.

Point one line too high and you get silence

Line 6 of Library.fs is the doc comment above double. There is no symbol there, so the answer is empty:

off by one, onto the comment
$ python3 tools/fsharp_lsp.py references tests/fixtures/SampleProject tests/fixtures/SampleProject/Library.fs 6 5
WITH config: 0 reference(s) in 0 file(s)
$ echo $?
1

Zero references means the position missed, and the exit code says so. The query includes the declaration, so a real symbol always returns at least itself — even dead code comes back as 1 reference. An empty answer is never “searched and found none”; it is a comment, a blank line or the wrong column, and it exits 1.

Wiring it into a script

Exit codes are 0 ok, 1 nothing found, 2 LSP or environment error — so a missed position fails a script rather than passing quietly through it. If fsautocomplete lives somewhere unusual, point at it with FSAC_PATH rather than reordering your PATH. Budget 2–6s warm and up to ~40s cold: each invocation starts a fresh server and loads the whole MSBuild graph, which is also why it can never serve a stale project graph.

/// troubleshooting

When it hangs, run doctor.

A hanging or silent LSP tool is almost always one of three things, and doctor checks all three by executing fsautocomplete rather than merely looking for it — the difference between a diagnosis and a hang. Here it is against a deliberately broken install:

a broken install
$ FSAC_PATH=/nonexistent/fsautocomplete python3 tools/fsharp_lsp.py doctor tests/fixtures/SampleProject
FAIL fsautocomplete not found at /nonexistent/fsautocomplete (from FSAC_PATH). Install it with 'dotnet tool install -g fsautocomplete', put the install directory on PATH, or set FSAC_PATH to the binary.
ok python 3.10
ok dotnet sdk 10.0.110
ok 1 restored project(s) under tests/fixtures/SampleProject
$ echo $?
2

It reports every check, not just the first failure — one run tells you the whole story, and the message names the fix.

⚠ The failure almost everyone hits

~/.dotnet/tools must be on the PATH of the process that launches Claude Code. A login shell that has it is not enough if your editor or launcher does not inherit that PATH — which is why install step 1 verifies the binary rather than trusting the install.

/// what it will not do

Read-only, for now.

  • ✗ It cannot write. No rename, no code actions, no formatting. A rename was built and worked, but it ran into enough open issues that it was deferred to a future version rather than shipped half-trusted; the code is still in the file, commented out, with notes on what it did. Until then the tool hands you the positions and step 4 stays yours. Formatting is Fantomas's job either way.
  • ? .fsi signature files and type providers are untested.
  • λ It is F#-only. The languageId is fixed; this is not a general LSP client.
  • ⏱ One server per CLI invocation. For casual navigation inside the current workspace, the plugin's warm server is faster.
